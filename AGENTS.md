# AGENTS.md

## Purpose

Operational guide for coding agents working in this repository.

## Project summary

`nix-user-repository` is the Nix packaging repo (NUR-style) for the Dirstral tools. It
packages prebuilt release binaries — nothing is built from source here. Each package
`fetchurl`s an upstream GitHub release tarball and installs the binary.

## Before you start

- Re-check that the request still matches the current tree before changing anything.
- Read `pkgs/<name>/default.nix` for the package you're touching; note the
  `# generated by GoReleaser. DO NOT EDIT.` header.
- Preserve the existing per-platform `urlMap`/`shaMap` structure and binary-fetch
  approach unless the task explicitly says otherwise.

## Repo map

- `pkgs/dir2mcp/default.nix` — dir2mcp package (prebuilt binary fetch). Only package so
  far; new tools follow the same `pkgs/<name>/default.nix` layout.
- `LICENSE` — MIT.
- No root `default.nix`, no `flake.nix`, no CI workflows currently.

## How a package is defined

- `stdenvNoCC.mkDerivation` with `src = fetchurl { url = urlMap.${system}; sha256 =
  shaMap.${system}; }`.
- `installPhase` copies the binary to `$out/bin`.
- `meta.sourceProvenance = [ lib.sourceTypes.binaryNativeCode ]`.
- Platforms: `x86_64-linux`, `aarch64-linux`, `x86_64-darwin`, `aarch64-darwin`.

## Generated files

`pkgs/dir2mcp/default.nix` is regenerated by upstream's GoReleaser on every release and
pushed here as an automated `dir2mcp: vX -> vY` commit. Do NOT hand-edit `version`,
`urlMap`, or `shaMap`. Packaging logic lives in the upstream `.goreleaser.*` config.

## Build / verify commands

```bash
# Build (callPackage-style file, no root aggregator):
nix-build -E 'with import <nixpkgs> {}; callPackage ./pkgs/dir2mcp {}'
./result/bin/dir2mcp version
```

A bad hash or dead URL fails the build; Nix prints the expected sha256.

## Git workflow

- Pull latest `main` before starting.
- Work on a branch; do not push directly to `main`.
- Use Conventional Commits (<https://www.conventionalcommits.org/>).
- Do not mention yourself or add co-author trailers in commit messages.

## Coding rules

- Work only in this repository unless told otherwise.
- Keep patches minimal and scoped.
- Never hand-edit generated version/URL/hash fields.
- Keep base32 (non-SRI) sha256 format if regenerating a hash.
- Do not add markdown/docs files unless explicitly requested.

## Review/merge readiness

- The changed package builds via `callPackage` and the binary runs.
- No unrelated files changed; no accidental flake/root-default.nix introduced.
- Generated headers and per-platform structure preserved.
